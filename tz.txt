Советы и рекомендации
Для конкурентного сценария проще всего начать с optimistic locking (@Version) — это минимальное изменение кода
При написании интеграционного теста на конфликт используйте CountDownLatch или CompletableFuture.allOf() для эмуляции параллельных запросов
Для тестов с Testcontainers не забудьте добавить зависимость и настроить application-test.properties
В Dockerfile используйте multi-stage сборку, если хотите уменьшить размер образа — но это не обязательно
При выборе инфраструктурных пунктов учитывайте их сложность: Security потребует больше времени, чем планировщик или кеширование


Критерии оценивания (максимум 40 баллов)
Обязательные условия допуска

Если хотя бы одно условие не выполнено, работа по этапу 2 не засчитывается, независимо от набранных баллов:

A) Docker: в репозитории есть Dockerfile и docker-compose.yml; проект поднимается командой docker compose up (приложение + PostgreSQL), миграции применяются.
B) Документация: README дополнен разделами этапа 2 (ссылка на Swagger, архитектурные решения, минимум 6 curl: 3 CRUD + 3 бизнес‑операции, инструкция запуска тестов).
C) Swagger доступен после запуска приложения.

1) Функциональность API и бизнес‑логика (12 баллов)
0 баллов — итоговые минимумы по API не выполнены (например, CRUD < 3 агрегатов или бизнес‑операций < 6 или отсутствуют 2 списочных endpoint’а), либо существенная часть API неработоспособна.
6 баллов — выполнено частично: минимумы достигнуты не полностью или есть заметные пробелы (например, часть CRUD/операций работает с ошибками; не везде есть DTO/валидация/коды 400/404/409; списки без полного набора требований).
12 баллов — выполнены все итоговые минимумы корректно:
- полный CRUD минимум для 3 агрегатов;
- минимум 6 бизнес‑операций (не CRUD), каждая: входной DTO + Bean Validation + бизнес‑правила + корректные коды 400/404/409;
- минимум 2 списочных endpoint’а, каждый: page/size, sort=field,asc|desc, фильтрация минимум по 3 параметрам (опциональные, применяются совместно).


2) Транзакции (6 баллов)
0 баллов — @Transactional отсутствует или используется некорректно (не обеспечивает атомарность/целостность).
3 балла — есть минимум 4 @Transactional метода, но multi-table сценарии выполнены частично (менее 2) или есть логические ошибки (частичные записи при ошибках, неверные границы транзакции).
6 баллов — выполнены требования:
- минимум 4 метода сервисов с @Transactional;
- минимум 2 сценария, где в одной транзакции изменяются несколько таблиц.

3) Конкурентный сценарий + интеграционный тест гонки (8 баллов)

8 баллов — сценарий реализован корректно и обоснован; есть воспроизводимый интеграционный тест (Testcontainers + PostgreSQL), демонстрирующий конфликт/гонку и корректную реакцию системы (например, 409 Conflict для optimistic/unique invariant).
